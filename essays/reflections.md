When building a building in real life, you need to start with a sturdy foundation to build higher. ICS 314 was that foundation for me. Before this semester (Spring 2021), I had pretty much no experience with things involving JavaScript, HTML, or anything about software development. Of all the courses I have taken so far, ICS 314 has to be the course that I am most fond of. It may appear to you that "ICS 314 - Software Engineering" is just about web application development, but there was so much more to it than that. This class gave us a chance to see how it would be like to code in a real-life environment and also gave experience on some fundamental software engineering concepts. These concepts can even be applied to other fields. Specifically, I will be looking at two concepts: Coding Standards and Agile Project Management.

## Coding Standards: Necessary!

Earlier in the semester I wrote about Coding Standards, which can be found by <a href="https://alvinedrada.github.io/essays/coding-standards.html">clicking here.</a> Basically, coding standards are similar to grammar rules in language, where there are a set of rules that you should follow when coding. By following coding standards, you help yourself and others to read your code better. If there was coding with no standards, you would waste a lot of time trying to understand. Having standards that have a common look and feel helps us be able to understand better in other projects. For example, this concept could be applied when writing an essay/paper or when at work where some jobs have certain standards that help make your job easier.

## Agile Project Management

<img class="ui small right floated image" src="../images/blueprint.jpg" alt="blueprint">

Agile project management is an approach to working on a project in parts to ensure continuous improvements of it. Issue Driven Project Management is a type of Agile Project Management, where it divides the project into milestones that is basically a deadline on when an certain task should be completed. The milestone is divided into issues, which helps to keep everything organized and helps hold whoever is assigned the issue accountable. An example of this could be seen in a project that I worked with a group on this semester, <a href="https://github.com/uhm-gitclubs/uhm-gitclubs/projects">UHM GitClubs</a>. I could see myself applying this to other things that involve a project with a group or even solo, at both school and work.
